## 已形成的主要工作链

### 1. 周报从“写文章”转为“证据工程”

- 数据源：飞书文档、会议/妙记、IM、日程、任务、OKR、本地 Codex Session、Git。
- 证据层：原始记录/工具读回 > 文件产物 > Session 最终总结 > 仅任务意图。
- 写作：一个判断、1–2 条主线、风险、下周重点；真人判断体现在取舍和依据，不是添加口头化语气。
- 发布：先草稿、再确认、后发布、最后读回；权限或来源缺失必须登记。

### 2. W10–W12 建立并行采集模式

- 独立采集文档、IM、会议/日程/任务、本地 Session。
- 采集 Agent 保持只读，主任务负责去重、互证、选择主线。
- 已发现的持续缺口：会议产物权限、文档/消息搜索 scope、附件/reactions 未全量读取。

### 3. Q2 OKR 收尾采用三条证据线

- OKR 系统状态：progress、indicator、status。
- 证据文档：本地补充材料与远端文档读回。
- API 限制：score 写入问题不能用 indicator/progress 成功替代说明。

### 4. 周报 Skill 和自动化边界已收敛

- `lark-weekly-report` 已创建并通过结构与压力测试。
- 真实生产链路仍待 W14 首次运行证明。
- 定时任务默认只生成草稿和覆盖审计，不自动发布。
- 飞书原生“汇报”记录暂不等同于普通文档生成/发布能力。

## Candidate reusable lessons

- Session 历史不能只看侧边栏最近列表；应按本机数据库 `cwd` 精确筛选，并保留统计口径。
- 并行 Agent 是证据采集单元，不应被误计为多个独立业务项目。
- 跨设备共享的是 Git 中的计划文件，不是本地 session binding；每台设备都需重新 attach。
